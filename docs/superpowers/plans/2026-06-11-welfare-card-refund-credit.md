# 福利卡退款退回账户实施计划

> **给智能执行者：** 必须使用 `superpowers:test-driven-development` 逐任务执行。步骤使用 checkbox (`- [ ]`) 跟踪。

**目标：** 退款成功回调时，把订单支付中的福利卡部分原路退回用户在销售加盟商下的福利卡账户，并写入退款流水。

**架构：** 复用现有退款回调事务。在首次 `succeeded` 回调、且原支付存在 `welfareCardPayableAmount > 0` 时，按订单 `salesFranchiseId` 与 `buyerUserId` 定位福利卡账户，增加余额，并写一条 `refund` 类型流水。重复 providerEventId 或已成功退款不重复退卡。

**技术栈：** NestJS API、Prisma、Jest、MySQL Docker 运行态。

**业务约束：**
- 主体关系固定为：平台 -> 加盟商 -> 商户 -> 商品/SKU -> 订单。
- 加盟商是福利卡发行方、真实销售方、销售账主体、收款归属主体。
- 商户是商品发布方、库存和履约方，并有实际地址；商户货款与加盟商结算。
- 用户支付方式为福利卡 + 线上现金支付组合支付；现金部分只允许微信支付或支付宝。
- 线下只存在加盟商与商户之间的结算打款确认，不属于用户支付方式。
- 平台没有核心门店/商店模型；`pickupStoreName` 只能作为历史兼容字段。

## 验收标准

- 首次退款成功回调：
  - 订单退款状态更新为 `succeeded`。
  - 福利卡账户余额增加原支付福利卡部分金额。
  - 福利卡流水新增 `type = refund`，金额为正数，包含 `orderNo` 和稳定幂等 `requestId`。
- 重复回调：
  - 返回 `duplicate: true` 或保持既有幂等语义。
  - 不重复增加福利卡余额。
  - 不重复写福利卡退款流水。
- 现金-only 订单退款：
  - 不访问福利卡账户。
  - 不写福利卡流水。
- 业务语义：
  - 退款归属销售加盟商下的用户福利卡账户。
  - 不引入线下现金支付。
  - 不引入门店/商店主链路。
