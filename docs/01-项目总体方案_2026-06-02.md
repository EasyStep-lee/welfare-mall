# 千店同城商城平台 V2 项目总体方案

## 1. 结论

新版项目应从空白仓库重建，不继续在旧系统上修补。旧系统只作为资源仓库：复用 UI、素材、历史数据和已验证业务规则，不复用旧字段关系、旧权限体系、旧 shop_id 主链路、旧 JSON 核心字段和旧结算边界。

新版的第一目标不是快速堆页面，而是先把业务底座做稳：

- 标准数据字典
- 统一身份与权限
- 加盟商和商户主体边界
- 商品主数据和商品池
- 订单、支付、退款、售后状态机
- 库存流水和履约任务状态机
- 加盟商和商户结算规则
- 审计、幂等、对账和回滚能力
- OpenAPI 接口契约和五端类型生成

## 2. 产品边界

### 保留五端

| 端 | 主要使用人 | 新版定位 |
|---|---|---|
| Admin 管理后台 | 平台内部员工 | 平台管理、审核、商品池、结算监管、风控、权限、审计 |
| Merchant 商户平台 | 商户团队 | 商品发布、SKU、库存、订单、售后、商户结算、资质 |
| Portal 门户网站 | 访客、企业客户、运营 | 门户展示、社区集采、商品池入口、申请入口 |
| User App 用户端小程序 | 用户 | 浏览、筛选、下单、支付、退款售后、福利卡、消息 |
| Logistics App 物流端小程序 | 配送和履约人员 | 任务列表、接单、取货、送达、异常、扫码核销、轨迹 |

后端不是一个前端端口，而是五端统一业务底座。

### 不进入 P0 的范围

- 原生 App
- 多级分销
- 复杂营销活动
- 大屏和复杂报表
- 自动补货
- 复杂物流调度
- 门店作为核心主体
- 线上自动分账

P0 必须保留未来扩展能力，但不把这些功能提前做进主链路。

## 3. 主体关系

新版保留平台、加盟商、商户、用户四类核心主体。门店不作为核心主体，不承担独立库存、履约、核销、结算责任。

```text
平台 -> 加盟商 -> 商户 -> 商品/SKU
用户 -> 订单 -> 加盟商/商户
```

旧字段处理：

- `shop_id`：历史兼容，只读保留，不进入新写入链路。
- `target_shop_id`：历史订单兼容，迁移后不再写入。
- `merchant_shop`：归档或转成商户附属地址、仓库、服务点。
- `local_store`：归档，不进入新版核心业务。

## 4. 技术架构

### 后端平台

后端采用 NestJS 模块化架构，按业务域拆分：

- `iam`：身份、角色、权限、数据范围、代理会话
- `franchise`：加盟商主体、区域、资质
- `merchant`：商户主体、账号、资质、经营资料
- `product`：商品主数据、SKU、规格、图片、资质
- `product-pool`：商品池、规则、渠道、快照
- `inventory`：库存、流水、扣减、回滚
- `order`：订单、订单状态机、订单金额
- `payment`：支付、回调、幂等、对账
- `refund`：退款、退款回调、冲账
- `aftersale`：售后申请、处理、凭证
- `fulfillment`：履约任务、配送状态、异常
- `settlement`：账单、结算单、调整、线下打款确认
- `asset`：文件素材、上传、引用关系
- `audit`：操作审计、状态变更日志
- `system-config`：字典、状态机配置、审批模板

核心约束：

- 金额统一用“分”。
- 核心字段结构化，不用 JSON 承载主业务关系。
- 支付、退款、库存、福利卡、结算必须有流水。
- 所有高风险操作必须审计。
- 所有前端通过 OpenAPI Client 调用接口，禁止手写漂移字段。

### 前端和小程序

Admin 和 Merchant 使用同一技术栈但保持独立应用、独立登录、独立权限边界。Portal 保留现有视觉结构，只替换新版商品池和装修数据来源。两个小程序继续走原生微信小程序路线，重点是重接新版 API 和移除 fallback 数据。

## 5. 数据模型优先级

第一批数据模型：

- 员工、部门、岗位、角色、权限、数据范围
- 加盟商、商户、区域
- 类目、品牌、规格属性模板
- 商品、SKU、商品媒体、商品资质
- 商品池、商品池规则、商品池渠道、商品池快照
- 文件素材库
- 审计日志、状态机配置、审批模板

第二批数据模型：

- 订单、支付、退款、售后
- 库存、库存流水、履约任务
- 加盟商账单、商户账单、结算单、打款记录
- 对账、调整、冲正

## 6. 旧资源迁移策略

旧资源先审计，再打标签：

- `KEEP`：可直接保留的素材、视觉、文案。
- `MIGRATE`：历史用户、订单、商品等需要清洗迁移的数据。
- `REWRITE`：业务逻辑正确但实现和字段不适合继承的功能。
- `ARCHIVE`：只保留历史查询价值的旧模块。
- `DROP`：不进入新版的门店主链路、本地服务模块、fallback 数据。

迁移不能从旧表直接映射到新版表。必须先形成新版 ER 图、数据字典、迁移映射和校验脚本，再导入。

## 7. 验收原则

每个模块上线前必须同时具备：

- 数据字典
- ER 图或关系说明
- OpenAPI 契约
- 权限矩阵
- 数据范围规则
- 状态机定义
- 审计规则
- 测试用例
- 迁移规则
- 回滚方案

验收分层汇报：

- 本地代码和自动化验证
- Docker/运行环境验证
- 五端联调验证
- 微信 DevTools 验证
- 真机验证
- 目标环境验证
- 业务人工验收

任何一个层级未执行，都不能合并成“全部完成”。
