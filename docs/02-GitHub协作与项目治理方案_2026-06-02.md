# GitHub 协作与项目治理方案

## 1. 仓库策略

建议采用一个主仓库承载新版项目，内部用 monorepo 管理后端和五端：

```text
apps/api
apps/admin
apps/merchant
apps/portal
apps/user-miniprogram
apps/logistics-miniprogram
packages/api-client
packages/contracts
packages/shared
docs
tools
```

理由：

- OpenAPI 契约、DTO、枚举和状态机可以在同一仓库中统一生成和验证。
- 五端联调、权限矩阵和环境门禁可以集中治理。
- 早期重构阶段跨模块变更多，monorepo 更容易保持接口一致。

## 2. 分支策略

| 分支 | 用途 |
|---|---|
| `main` | 稳定主线，只接受通过验证的 PR |
| `codex/<topic>` | Codex 或自动化执行分支 |
| `feat/<topic>` | 人工功能分支 |
| `fix/<topic>` | 缺陷修复分支 |
| `release/<version>` | 预发布收口分支 |

规则：

- 不直接在 `main` 开发。
- 每个 PR 只对应一个可验收主题。
- 需求、实现、验收证据和回滚方案必须在 PR 中闭环。
- 涉及支付、退款、库存、结算、权限的 PR 必须附带定向测试和全量门禁结果。

## 3. GitHub Project 看板

建议建立一个 GitHub Project：`千店同城商城平台 V2`。

字段：

- `Area`：API、Admin、Merchant、Portal、User App、Logistics App、Data、DevOps、Docs
- `Domain`：IAM、Merchant、Franchise、Product、Product Pool、Order、Payment、Refund、Inventory、Fulfillment、Settlement、Audit
- `Phase`：P0、P1、P2、P3
- `Acceptance Layer`：Local、Docker Runtime、Five-end Integration、DevTools、Real Device、Target Environment、Business Signoff
- `Risk`：Low、Medium、High、Critical
- `Owner`：负责人

列：

- `Inbox`
- `Ready`
- `In Progress`
- `Code Complete`
- `Local Verified`
- `Runtime Verified`
- `Target Pending`
- `Business Acceptance`
- `Done`
- `Blocked`

注意：`Code Complete` 不等于 `Done`。真机、目标环境和业务验收需要单独列。

## 4. Issue 体系

Issue 类型：

- `Epic`：阶段级大目标，例如 P0 基础内核。
- `Story`：可交付业务能力，例如商品池发布规则。
- `Acceptance`：验收任务，例如五端商品详情契约联调。
- `Bug`：缺陷。
- `Tech Debt`：技术债。
- `Decision`：架构决策。
- `Migration`：旧资源迁移。

每个 Story 必须包含：

- 背景
- 范围
- 非范围
- 数据模型影响
- 接口契约影响
- 权限和数据范围
- 状态机或审计影响
- 验收层级
- 回滚方式

## 6. Milestone 建议

### P0-基础内核

交付：

- monorepo 基础结构
- 后端 NestJS + Prisma + MySQL + Redis 基础
- OpenAPI 契约生成
- IAM、加盟商、商户、区域、商品基础模型
- 商品池基础模型
- 审计和状态机基础能力

### P1-商品池与五端接入

交付：

- 商品发布与审核
- 商品池规则和渠道
- Admin 商品工作台
- Merchant 商品发布
- Portal 商品池展示
- User App 商品浏览和详情

### P2-交易履约结算

交付：

- 订单、支付、退款、售后
- 库存扣减和回滚
- 履约任务和物流端任务流
- 加盟商账单、商户账单、结算单
- 幂等、对账、冲正

### P3-迁移上线准备

交付：

- 旧资源审计和迁移包
- 环境变量模板
- 目标环境部署
- 五端联调
- 真机验收
- 业务验收
- 回滚演练

## 7. PR 门禁

每个 PR 必须填写：

- 改动范围
- 业务影响
- 数据库迁移影响
- 权限影响
- 支付、退款、库存、结算影响
- 验证命令和结果
- 截图或运行证据
- 未覆盖风险
- 回滚方式

合并条件：

- CI 通过
- 关键测试通过
- OpenAPI 契约无漂移
- 数据库迁移可回滚或有前向修复方案
- 高风险模块至少一名人工复核

## 8. GitHub Actions 门禁

早期先建立文档和治理门禁，代码进入后逐步扩展：

- 文档完整性检查
- lint
- typecheck
- unit tests
- integration tests
- OpenAPI contract generation
- Prisma migration validation
- Playwright smoke tests
- mini-program release config checks
- Docker runtime smoke tests

高风险模块必须保留定向门禁：

- 支付回调幂等
- 退款回调幂等
- 福利卡流水
- 库存扣减和回滚
- 订单状态机
- 账单生成和冲正
- 权限矩阵
- 数据范围
