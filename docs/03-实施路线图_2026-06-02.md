# 实施路线图

## 阶段 0：项目治理和基础资料固化

目标：先把“怎么做、按什么验收、什么不能继承”固定下来。

交付物：

- 项目总体方案
- GitHub 协作方案
- 业务规则与结算口径基线
- 数据字典目录
- ER 图目录
- 状态机目录
- 权限矩阵目录
- 旧资源审计清单模板

验收：

- GitHub Project、Labels、Milestones、Issue 模板、PR 模板建立。
- 所有 P0 需求进入 Issue。
- 旧系统只作为资源仓库的边界写入 ADR。

## 阶段 1：后端基础内核

目标：建立可支撑五端的统一业务平台。

顺序：

1. monorepo 和基础工具链
2. NestJS API 应用
3. MySQL 8、Prisma、Redis、Docker
4. OpenAPI 文档和 Client 生成
5. IAM 基础：用户、部门、岗位、角色、权限、数据范围
6. 加盟商、商户、区域基础模型
7. 审计日志和代理会话基础
8. 状态机配置基础

验收：

- 单元测试通过。
- 集成测试能跑通数据库和 Redis。
- OpenAPI 能生成前端 Client。
- Admin 和 Merchant 能基于权限返回不同菜单和数据范围。

## 阶段 2：商品主数据和商品池

目标：先做商品和商品池，再接页面。

顺序：

1. 类目、品牌、规格属性模板
2. 商品、SKU、规格、媒体、资质
3. 商品发布草稿
4. Admin 商品审核
5. 商品池、商品池规则、商品池渠道、商品池快照
6. Merchant 商品发布入口
7. Portal 商品池展示
8. User App 商品浏览和详情

验收：

- 商品名称、价格、图片、SKU、库存真源统一。
- 运营不能在装修里手改商品名称、价格、主图。
- 前端不展示内部 ID 输入。
- 商品详情接口由 OpenAPI 契约生成并被五端消费。

## 阶段 3：订单、支付、退款、售后

目标：交易主链路先闭环，再接复杂运营。

顺序：

1. 订单状态机
2. 下单金额计算
3. 福利卡和现金支付金额拆分
4. 支付单和支付回调幂等
5. 退款单和退款回调幂等
6. 售后申请和处理
7. 订单取消、关闭、完成
8. User App 下单、支付、退款售后
9. Admin 客服和财务监管

验收：

- 重复支付回调不重复入账。
- 重复退款回调不重复冲账。
- 金额关系可验证。
- 每次状态变化有审计记录。

## 阶段 4：库存、履约、物流端

目标：库存和履约必须依赖后端状态机，不由前端推断。

顺序：

1. 库存账户和库存流水
2. 下单扣减和取消回滚
3. 支付后履约任务生成
4. 物流端任务列表和任务详情
5. 接单、取货、配送中、送达
6. 异常上报
7. 扫码核销
8. 轨迹和位置

验收：

- 库存扣减和回滚都有流水。
- 履约状态只能按允许路径流转。
- 扫码核销依赖后端核销结果。
- 真机和目标环境验收单独记录。

## 阶段 5：结算、对账、打款确认

目标：结算可追溯、可冲正、可人工确认，先不做线上自动分账。

顺序：

1. 加盟商账单
2. 商户账单
3. 退款冲减账单
4. 补偿和人工调整
5. 结算单生成
6. 结算单锁定
7. 线下打款确认
8. 冲正
9. 对账报表

验收：

- 支付成功生成正向账单。
- 退款成功生成冲减账单。
- 结算单锁定后不能直接修改原账单。
- 人工调整必须有原因、审批和审计。

## 阶段 6：旧资源迁移和上线准备

目标：迁移只迁移经过清洗和映射的数据，不继承旧结构。

顺序：

1. 旧资源审计
2. KEEP / MIGRATE / REWRITE / ARCHIVE / DROP 打标
3. 新旧字段映射
4. 数据清洗
5. 迁移脚本
6. 迁移校验
7. 回滚演练
8. 五端联调
9. 目标环境验收
10. 上线审批

验收：

- 迁移前后关键金额、订单、用户、商品数量可对账。
- 目标环境验证和人工验收有独立记录。
- 回滚方案经过演练。
