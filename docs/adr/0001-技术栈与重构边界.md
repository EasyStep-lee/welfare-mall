# ADR-0001：技术栈与重构边界

## 状态

Accepted

## 背景

旧系统功能覆盖较多，但存在主体边界混乱、shop_id 渗透、商品详情字段不统一、JSON 核心字段过多、权限边界不清、结算链路需要重定义、数据库弱约束、小程序和门户存在 fallback 数据等问题。

继续修补旧系统会把技术债带入新版。新版必须以旧系统为资源仓库，重新建立业务内核。

## 决策

新版采用以下技术栈：

- 后端平台：NestJS + TypeScript + MySQL 8 + Prisma + Redis + OpenAPI + Jest + Docker
- Admin：Vue 3 + TypeScript + Vite + Pinia + Element Plus + OpenAPI Client + Playwright
- Merchant：Vue 3 + TypeScript + Vite + Pinia + Element Plus + OpenAPI Client + Playwright
- Portal：Vue 3 + TypeScript + Vite + 静态生成/预渲染 + OpenAPI Client
- User App：原生微信小程序 + TypeScript + 统一 API Client + 分包
- Logistics App：原生微信小程序 + TypeScript + 统一 API Client + 地图/扫码能力

重构边界：

- 复用 UI，不复用旧数据模型。
- 复用素材，不复用旧字段关系。
- 复用成熟业务经验，不复用旧架构。
- 复用支付退款能力，不复用旧结算边界。
- 旧系统作为资源仓库，不作为新版架构标准。

## 影响

- P0 不引入跨端框架。
- P0 不开发原生 App，但保留 API 可接入能力。
- 门店不作为新版核心主体。
- 商品池成为商品分发中心。
- Admin、Merchant、Partner 入口隔离。
- 平台人员协助加盟商或商户时使用代理会话并审计。

## 验收

- 新版代码不得依赖旧 shop_id 主链路。
- 新版核心业务字段必须结构化。
- 五端接口必须通过 OpenAPI 契约生成和校验。
- 支付、退款、库存、结算必须具备幂等和流水。
- 本地、运行环境、真机、目标环境和业务验收必须分层汇报。
